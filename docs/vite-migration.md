# Vite Migration Guide for This Project (Flask + Vue)

This document explains how to migrate this project from CDN-based frontend assets to a proper Vite build pipeline.

It is written specifically for this repository, where Flask serves templates and APIs while Vue currently runs from static JavaScript.

## 1) Current State and Why Migrate

Current frontend pattern:
- HTML is rendered by Flask templates.
- Vue, Axios, Socket.IO, and icons are loaded via CDN in templates.
- Main client logic lives in a single static JavaScript file.

Why migrate to Vite:
- Better dependency management (locked versions, no random CDN drift).
- Faster local development and hot reload.
- Production-ready bundling, minification, and cache-safe hashed assets.
- Easier testing and gradual modularization.
- Cleaner deployment model and easier rollback.

## 2) Recommended Architecture

Use this split:
- Backend: Flask remains the owner of routes, APIs, WebSocket events, templates.
- Frontend build: Vite compiles JavaScript/CSS into static assets.
- Template rendering: Flask injects built asset URLs from Vite manifest.

Suggested folder layout:

  sorot/
  - frontend/
    - src/
      - main.js
      - styles.css
    - package.json
    - vite.config.js
  - static/
    - dist/               (Vite build output)
  - templates/
    - index.html
  - sorot.py

## 3) Migration Strategy (Safe and Incremental)

Use a phased migration to reduce risk.

Phase 1: Build pipeline only
- Keep existing Flask templates and DOM structure.
- Move frontend JS/CSS into frontend/src.
- Bundle with Vite and serve built files from Flask.

Phase 2: Dependency cleanup
- Replace CDN global dependencies with npm dependencies.
- Switch global calls to imports where practical.

Phase 3: Modularization
- Split large app file into feature modules.
- Add unit tests for utilities and critical logic.

Phase 4 (optional): Full Vue SFC architecture
- Move from in-template Vue markup to component files if needed.

## 4) Step-by-Step Implementation

## 4.1 Create frontend workspace

From repo root:

  mkdir frontend
  cd frontend
  npm init -y
  npm install vue axios socket.io-client lucide
  npm install -D vite

Then create scripts in frontend/package.json:
- dev: vite
- build: vite build
- preview: vite preview

## 4.2 Create Vite config

Create frontend/vite.config.js with these goals:
- Build output goes to ../static/dist
- Keep manifest enabled for Flask lookup
- Configure dev proxy to Flask backend

Example config shape:

  import { defineConfig } from 'vite'
  import path from 'path'

  export default defineConfig({
    root: '.',
    server: {
      port: 5173,
      strictPort: true,
      proxy: {
        '/api': 'http://127.0.0.1:5000',
        '/socket.io': {
          target: 'http://127.0.0.1:5000',
          ws: true
        },
        '/upload': 'http://127.0.0.1:5000',
        '/downloaded': 'http://127.0.0.1:5000'
      }
    },
    resolve: {
      alias: {
        // Needed when Vue templates are compiled at runtime from DOM/template files
        vue: 'vue/dist/vue.esm-bundler.js'
      }
    },
    build: {
      outDir: path.resolve(__dirname, '../static/dist'),
      emptyOutDir: true,
      manifest: true,
      rollupOptions: {
        input: {
          app: path.resolve(__dirname, 'src/main.js')
        }
      }
    }
  })

## 4.3 Move frontend entry code

Create frontend/src/main.js.

Initial low-risk approach:
- Import dependencies and expose them on window for compatibility.
- Import your existing app logic.

Pattern:

  import { createApp } from 'vue'
  import axios from 'axios'
  import { io } from 'socket.io-client'
  import lucide from 'lucide'
  import './styles.css'

  window.Vue = { createApp }
  window.axios = axios
  window.io = io
  window.lucide = lucide

  import '../../static/js/app.js'

Later, refactor app.js to standard imports and remove window globals.

## 4.4 Move CSS into frontend source

Options:
- Quick: import ../../static/css/app.css from frontend/src/main.js.
- Better: copy app.css into frontend/src/styles.css and keep all frontend style assets inside frontend/src.

## 4.5 Update Flask template loading

In template files:
- Remove CDN script tags for Vue/Axios/Socket.IO/Lucide.
- Remove direct script include of static/js/app.js.
- Include built files generated by Vite.

Use Flask helper to read static/dist/.vite/manifest.json and inject the JS and CSS assets.

## 4.6 Add Flask helper for Vite manifest

Add a utility in sorot.py that:
- Reads static/dist/.vite/manifest.json.
- Resolves an entry key like src/main.js.
- Returns JS file path and CSS list.
- Falls back gracefully in development or when missing.

Then pass those asset paths into render_template for index.

## 4.7 Development modes

Use two terminals:
- Terminal A: run Flask backend on 5000.
- Terminal B: run Vite dev server on 5173 from frontend.

Two development options:
- Option A (recommended): use Vite only for frontend preview and proxy backend calls.
- Option B: keep Flask page rendering and point script tags to Vite dev URL while in debug mode.

Option A is simpler at first. Option B gives one URL for page rendering.

## 4.8 Production build flow

Before deployment:
- npm run build inside frontend.
- Ensure static/dist contains hashed assets and manifest.
- Start Flask app.

Flask serves built static assets directly.

## 5) Robustness Checklist

Use this checklist after migration:

Dependency safety:
- package-lock.json is committed.
- Frontend dependency versions are pinned.
- No critical runtime dependency loaded from CDN.

Build stability:
- npm run build works on clean machine.
- Manifest exists and is readable by Flask.
- Flask fails gracefully with useful error if manifest missing.

Runtime behavior:
- Main page loads without console errors.
- Socket connection works in both dev and production.
- API calls use relative paths and work through proxy.

Performance:
- Large vendor chunks are acceptable or split if too large.
- Compression enabled at reverse proxy layer for production.

Security:
- No secret values exposed in frontend bundle.
- CORS and Socket.IO origins are restricted for production.

Observability:
- Basic frontend error logging exists.
- Backend logs include build/manifest loading diagnostics.

## 6) Testing Plan

Minimum tests to run each change:
- Backend tests already in tests/ still pass.
- Manual smoke test for mode selection, upload/import, recording, and reports.
- Verify settings page remains intact.

Suggested additions:
- Small frontend utility tests (frame conversion, ROI overlap checks).
- Playwright or Cypress smoke test for main happy path.

## 7) Common Migration Pitfalls

1. Vue runtime build mismatch
- Symptom: template compilation errors.
- Fix: alias vue to vue/dist/vue.esm-bundler.js in Vite config.

2. Socket.IO broken in dev
- Symptom: websocket 404 or failed upgrade.
- Fix: add /socket.io proxy with ws true in Vite config.

3. Flask cannot find built files
- Symptom: blank page or missing script/css.
- Fix: verify manifest path and entry key names.

4. Cache confusion after deployment
- Symptom: stale UI after update.
- Fix: use hashed assets via manifest and avoid hardcoded asset filenames.

5. Mixed absolute URLs
- Symptom: API works in one mode only.
- Fix: use relative API paths and centralize base URL handling.

## 8) Suggested CI/CD Workflow

Pipeline steps:
1. Install Python dependencies.
2. Install Node dependencies in frontend.
3. Build frontend with npm run build.
4. Run Python tests.
5. Archive/deploy with static/dist included.

This guarantees every deployment ships a known frontend build.

## 9) Rollback and Recovery

Keep one safe fallback during transition:
- Preserve current static/js/app.js path and old template include behind a feature flag.
- If Vite build fails in production, fall back to old assets temporarily.

Once stable for several releases, remove fallback path.

## 10) Definition of Done

Migration is complete when:
- CDN runtime dependencies are removed from main app page.
- Frontend builds through Vite and assets are hashed.
- Flask loads assets via manifest in production.
- Local dev flow is documented and repeatable.
- Smoke tests and core test suite pass.

## 11) Practical Next Action for This Repo

If you want to execute this immediately, start with this exact sequence:
1. Create frontend package and Vite config.
2. Add frontend/src/main.js that boots current app with compatibility globals.
3. Build to static/dist and wire manifest helper in sorot.py.
4. Update templates/index.html to use Vite-built assets.
5. Verify upload, scene editing, ROI drawing, recording, post-process, and report generation.

This gives you a robust foundation now, while allowing gradual cleanup later.
